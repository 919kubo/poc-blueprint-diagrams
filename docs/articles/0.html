# DXは技術の話ではありません ## ― 判断できる構造を作るということ DXという言葉を聞くと、 AIやクラウド、自動化といった技術を思い浮かべる方が多いと思います。 もちろん、それらはDXの一部です。 ただし、それだけではDXは成功しません。 現場でよく起きているのは、 **「新しい仕組みを入れたのに、判断は楽になっていない」** という状態です。 ## DXの失敗は、判断が曖昧なことから始まります DXがうまくいかない理由は、 ツール選定やシステムの性能不足ではありません。 多くの場合、次のことが決まっていないまま進んでいます。 - どこまでを仕組みに任せるのか - どこからは人が判断するのか - その判断の責任は誰が持つのか これが曖昧なままでは、 AIを使っても、業務はかえって複雑になります。 ## DXとは、判断できる構造を設計することです このサイトでは、DXを次のように考えています。 **DXとは、判断できる構造を設計することです。** 判断を減らすのか、 判断を固定するのか、 判断を人に残すのか。 その境界を先に決めておくことが、 DXの出発点になります。 ## なぜPMOの話をするのでしょうか PMOというと、 進捗管理や会議運営のイメージを持たれるかもしれません。 しかし、本来のPMOの役割は、 人とシステム、開発と運用のあいだに立ち、 **判断がきちんと行える状態を保つこと**です。 PMOは管理者というより、 判断が壊れないようにするための「設計役」だと考えています。 ## この先で扱うテーマ このシリーズでは、次のようなテーマを扱っていきます。 - 判断の境界はどこに引くべきか - AIに任せてよいこと、任せてはいけないこと - なぜ古い仕組みは残り続けるのか - 人とAIはどう協力すべきか DXを「流行の言葉」で終わらせず、 **判断を楽にする仕組みとして理解する**ための整理です。 次の記事では、 DXに欠かせない「境界」の考え方からお話しします。
